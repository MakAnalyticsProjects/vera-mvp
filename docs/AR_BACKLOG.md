## P0 — Live data on the dashboard (the only thing we're working on next)

### What

Move both the metrics pages **and** the Write-offs page off of bundled JSON snapshots and onto live, DB-backed reads. After this lands, every successful backfill (Rooflink jobs and line items) is reflected on the dashboard without a deploy.

This is the only piece of work in scope. P1/P2 items from the prior draft of this doc are deferred until P0 is in production.

### Why

Two complaints today, both rooted in the same gap:

1. **Israel's "metrics don't update on sync"** — the five existing pages (Aging, Milestones, Follow-ups, Rep Leaderboard, Reconciliation) read `apps/web/data/generated.json`, a static file built once at deploy time. Backfills update the DB; they do not move what the operator sees.
2. **Write-offs is the same architecture** — it ships from `apps/web/data/write-offs.json`, generated by `scripts/fetch-write-offs.ts` and committed. New Amount Withheld discounts in Rooflink won't surface until someone re-runs the script and deploys.

The DB infrastructure for both is already built (`RawRooflinkJob`, `RawRooflinkLineItems`, atomic-promote `BackfillRun`, `merge-view.ts`). What's missing is the read-path wiring.

### Scope — one cutover, two surfaces

**Metrics path (jobs):**

- Replace `getData()` in `apps/web/lib/data.ts` with a DB-backed loader: `getLiveJobs(tenantId)` from [merge-view.ts](../apps/web/lib/backfill/merge-view.ts) → `toARJob()` from `@vera/domain` at request time. Same transform the build-time preprocess uses today.
- Thread `tenantId` into the five API routes (`/api/jobs/aging`, `milestones`, `follow-ups`, `reconciliation`, `/api/reps/outstanding`). Auth context already populates it.
- Replace pinned `now = '2026-05-05'` from the preprocess with `new Date()` at request time. Verify aging buckets and heat scores still look sane before flipping.

**Write-offs path (line items):**

- Extract the write-off-detection logic from [scripts/fetch-write-offs.ts](../scripts/fetch-write-offs.ts) into a pure `shared/domain/write-offs.ts` module: filter `discounts[]` by `product_id === 71493`, map to the `WriteOffRecord` shape. Both the script and the new DB route should call the same function.
- Replace `getWriteOffs()` in `apps/web/lib/write-offs-data.ts` with a DB-backed loader: `getLiveLineItems(tenantId)` from `merge-view.ts` → the new domain function → `WriteOffRecord[]`. Roll up totals at request time.
- Wire `/api/jobs/write-offs` to the new path.

**Shared:**

- Cache the computed snapshot per promoted-run-id (in-process Map first; Vercel Runtime Cache later if needed). Cache key spans both sources — bumping either run promotes a fresh cache slot.
- Single feature flag (`USE_DB_DATA_SOURCE=1`) gates both metrics and write-offs together so rollback is one toggle.

### Dependencies (must happen first)

The DB is not seeded with real Rooflink data today — every promoted `BackfillRun` in Neon is a mock fixture. Cutover cannot ship until:

| Step | Time | Notes |
|---|---|---|
| Real `rooflink_jobs` backfill, promoted | ~17 min @ 1 req/sec for 103,440 jobs | Can be slower under Rooflink WAF throttle. |
| Real `rooflink_lineitems` backfill, promoted | ~2.5 hr @ 1 req/sec for ~8,500 estimates | Depends on the jobs run completing first — line-items walks estimate IDs from the promoted job snapshot. |

Both can run in the background. The dashboard stays on the JSON path until the flag flips.

### Effort

2–3 focused days once the real backfills have seeded the DB. Breakdown:

- ~half day — DB-backed `getData()` + tenant threading + 5 route updates
- ~half day — write-off domain extraction + DB-backed `getWriteOffs()` + route update
- ~half day — caching layer + feature flag + invalidation on promote
- ~half day — before/after verification, regression sweep, e2e updates, deploy
- ~half day — buffer

### Risks worth flagging

- **"Today" stops being May 5, 2026.** Every aging bucket and heat score recomputes against the real current date when the flag flips. Show Israel a before/after on a sample row before the cutover so the shift doesn't look like a bug.
- **Anomaly detection is O(n²)** in the worst case. Today it runs once at build time; in the DB path it runs per request (cached). For ~12k AR jobs this should be sub-100ms, but worth measuring.
- **Multi-tenant scoping.** The schema joins tenant via `BackfillRun.tenantId`; the merge view already handles this. Sanity-check that `getLiveJobs(tenantId)` and `getLiveLineItems(tenantId)` never leak rows across tenants once a second tenant exists.

Full architectural plan and step-by-step migration table in [docs/DATA_SOURCE_MIGRATION.md](DATA_SOURCE_MIGRATION.md). That doc covers the metrics path in detail; the write-offs additions above extend it.
